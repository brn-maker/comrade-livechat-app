Live Chat MVP: Architectural Plan for CursorThis document serves as the master roadmap for Cursor's Agent Mode to build a working MVP of a filtered live video chat platform. The focus is on a "site-first" prototype to validate demand for gender-based matching and AR filters.Phase 1: Environment & Core DependenciesObjective: Initialize the Next.js project and install the necessary real-time and AI libraries.Initialize Project: Create a Next.js app (App Router, TypeScript, Tailwind CSS).Install Communication Libraries: socket.io, socket.io-client, and simple-peer (or raw WebRTC wrappers).Install Data Libraries: @supabase/supabase-js (Auth/Profiles) and @upstash/redis (Matching Queues).Install AR/AI Libraries: @mediapipe/tasks-vision for client-side face landmarking.Phase 2: User Foundation (Supabase)Objective: Set up the persistent data layer for user declarations.Supabase Schema: Create a profiles table:id (UUID, primary key)declared_gender (text: 'male', 'female', 'other')birth_year (int)interested_in (text: 'male', 'female', 'both')Auth Integration: Use Cursor to generate a simple "Sign Up / Guest Login" flow using Supabase Auth.Declaration Form: Build a high-conversion landing page component where users declare these traits before entering the chat pool.Phase 3: Real-Time Engine (Socket.io & Upstash Redis)Objective: Build the "Intent-Based" matching engine.Signaling Server: Set up a Node.js server (standalone or via Next.js custom server) to coordinate WebRTC handshakes.Redis Queue Logic: Use Redis Sorted Sets (ZSETs) for matchmaking:Key: queue:[userGender]_seeking_[targetGender] (e.g., queue:male_seeking_female).Score: Date.now() (Ensures FIFO matching).Matching Algorithm:When a user joins, check the corresponding "seeking" queue.If a match is found, "pop" both users and emit a match_found event via Socket.io with a unique roomID.If no match, add the user to their own queue.Role Assignment: Assign initiator and responder roles at match time to prevent WebRTC race conditions.Phase 4: WebRTC Video & AR Filter PipelineObjective: Implement the processed video stream with face filters.Media Capture: Access local webcam via navigator.mediaDevices.getUserMedia.The Canvas Intermediary:Render the raw webcam feed to a hidden <canvas>.Initialize MediaPipe Face Landmarker in LIVE_STREAM mode with GPU acceleration.Filter Implementation:Draw the video frame to the canvas.Overlay AR effects (e.g., beauty filters, masks) based on the 478 detected landmarks.Stream Transmission: Use canvas.captureStream() to turn the filtered frames into a MediaStream for the WebRTC connection.Track Swapping: Use RTCRtpSender.replaceTrack() to allow users to change filters mid-stream without dropping the connection.Phase 5: Frontend UI & Ad IntegrationObjective: Build the user interface and initial monetization containers.Chat Workspace: Build a dual-video layout using Tailwind CSS:Remote user (large)Local user with active filter (small picture-in-picture).Control Bar: Add prominent "Start," "Skip / Next," and a "Filter Gallery" carousel.Ad Placements:Top: 728x90 Leaderboard container.Sidebar: 300x250 Rectangle container.Integrated "Social Bar" ad unit (small notification-style overlay).State Management: Display "Searching for [Gender]..." status updates to manage wait-time expectations.Phase 6: Analytics & Proving DemandObjective: Collect data to justify the transition to premium subscriptions.Skip Tracking: Log session lengths to Supabase. Compare session duration for "filtered" matches vs. "random" matches to prove value.Manual Feedback: Add a simple "Report" button to manually log NSFW incidents for future automated moderation planning.Wait-Time Logic: Implement a basic fallback—if no filtered match is found within 30 seconds, offer to "Widen search" or "Go Random" to prevent churn.Deployment ChecklistFrontend: Deploy to Vercel.Backend/Sockets: Deploy to a platform supporting persistent WebSockets (e.g., Railway, DigitalOcean, or Render).Redis: Connect the production Upstash Redis instance.
